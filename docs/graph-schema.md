# Graph schema reference

This document is the user-facing reference for the Weld connected-structure
graph written to `.weld/graph.json`. It describes the file layout, the
closed vocabulary of node and edge types, the provenance and confidence
fields, the determinism and schema-version guarantees, and the rules for
extending the vocabulary.

Source of truth for the constants lives in:

- `weld/contract.py` -- `SCHEMA_VERSION`, `VALID_NODE_TYPES`,
  `VALID_EDGE_TYPES`, and all closed-vocabulary prop sets.
- `weld/_contract_validators.py` -- the `validate_graph`,
  `validate_node`, `validate_edge`, and `validate_fragment` validators
  surfaced by `wd graph validate` / `wd graph validate-fragment`.
- `weld/serializer.py` -- the canonical emitter that every writer of
  `graph.json` must go through.
- `weld/_graph_schema.py` -- federation schema-version gating.

Related ADRs:

- `docs/adrs/0011-polyrepo-federation.md` -- polyrepo federation
  (`repo:*` nodes, cross-repo edges).
- `docs/adrs/0012-determinism-contract.md` -- determinism contract and
  schema versioning.
- `docs/adrs/0016-edge-type-vocabulary-extension.md` -- governance and
  provenance edges (`owned_by`, `gates`, `supersedes`, `validates`,
  `generates`, `migrates`, `contracts`).
- `docs/adrs/0017-graph-freshness-source-file-model.md` -- graph
  freshness and the `meta.git_sha` stamp.
- `docs/adrs/0021-agent-graph-schema-vocabulary.md` -- Agent Graph node
  and edge vocabulary.
- `weld/docs/adr/0004-call-graph-schema-extension.md` -- symbol nodes
  and `calls` edges.

For the ROS2 and interaction-surface prop vocabularies, `weld/contract.py`
is the canonical source of truth. The inline docstrings in that module
cite the deciding ADRs.

## Location and top-level shape

The graph is stored at `.weld/graph.json` at the root of a Weld-initialized
repository or workspace. It is written atomically through
`weld.serializer.dumps_graph` and is regenerated by `wd discover`.

The file is a single JSON object with three required keys:

```json
{
  "meta": { ... },
  "nodes": { "<node-id>": { ... }, ... },
  "edges": [ { ... }, ... ]
}
```

Unknown top-level keys are preserved verbatim by the serializer for
forward compatibility but are not validated by the contract.

## `meta` block

The `meta` block carries the contract version, the generation timestamp,
the federation schema version, and optional freshness metadata.

| Field | Required | Type | Description |
|---|---|---|---|
| `version` | yes | int | Contract vocabulary version. Must equal `weld.contract.SCHEMA_VERSION` (currently `5`). A mismatch causes `validate_graph` to emit a remediation message that names the current version and the `wd discover` invocation needed to regenerate. |
| `updated_at` | yes | string | ISO-8601 timestamp written by the emitter on every save. |
| `schema_version` | written on save | int | Federation layout version: `1` for a single-repo or child graph, `2` for a federated root graph that contains `repo:*` nodes. See ADR 0011 and ADR 0012 for the gating rules. |
| `git_sha` | optional | string | HEAD commit SHA stamped by `Graph.save(touch_git_sha=True)` when the root is a git working tree. ADR 0017 describes how this participates in `wd stale`. |

`meta.version` is the **contract** version (what node/edge types exist).
`meta.schema_version` is the **layout** version (whether the file carries
federation constructs). They are orthogonal.

## Nodes

`nodes` is an object keyed by node id. Every entry has the following
shape:

```json
"<node-id>": {
  "type": "<node-type>",
  "label": "<human-readable label>",
  "props": { ... }
}
```

### Required node fields

| Field | Type | Description |
|---|---|---|
| `type` | string | Must be drawn from the closed set `VALID_NODE_TYPES`. See the vocabulary tables below. |
| `label` | string | Human-readable display name. |
| `props` | object | Metadata bag. May be empty. |

Referential-integrity of edges depends on node ids being unique strings;
the JSON object form already guarantees this.

### Node type vocabulary

The set below reflects `VALID_NODE_TYPES` as of contract version 5. New
types may be added only under the process in "Extending the schema"
below.

Core domain model:

- `service`, `package`, `entity`, `stage`, `concept`, `doc`, `route`,
  `contract`, `enum`, `file`, `dockerfile`, `compose`
- `agent`, `command`, `tool`, `workflow`, `test-suite`, `config`,
  `policy`, `runbook`
- `build-target`, `test-target`, `boundary`, `entrypoint`, `gate`,
  `deploy`
- `symbol` -- function-level callable. See
  `weld/docs/adr/0004-call-graph-schema-extension.md`.

ROS2 vocabulary:

- `ros_package`, `ros_interface`, `ros_node`, `ros_topic`,
  `ros_service`, `ros_action`, `ros_parameter`.

Generalized interaction surfaces:

- `rpc` -- request/response or stream method exposed or consumed by a
  module.
- `channel` -- named pub/sub or stream endpoint.

Agent Graph (see ADR 0021):

- `subagent`, `skill`, `instruction`, `prompt`, `hook`, `mcp-server`,
  `permission`, `platform`, `scope`.

Polyrepo federation:

- `repo` -- one `repo:<name>` node per registered child in the root
  meta-graph. Presence of any `repo:*` node triggers
  `meta.schema_version = 2` on save. See ADR 0011.

### Optional node props

The validator accepts the following optional fields on `props`. Any
additional prop is tolerated and round-tripped.

| Prop | Type | Vocabulary / notes |
|---|---|---|
| `source_strategy` | string | Name of the discovery strategy or agent that emitted the node. Primary provenance signal. |
| `authority` | string | One of `canonical`, `derived`, `manual`, `external`. |
| `confidence` | string | One of `definite`, `inferred`, `speculative`. |
| `roles` | list[string] | Subset of `implementation`, `test`, `config`, `doc`, `build`, `migration`, `fixture`, `script`. |
| `file` | string | Source file path relative to the repository root. |
| `span` | object | `{ "start_line": int, "end_line": int }`; `start_line <= end_line`. |
| `doc_kind` | string | One of `adr`, `policy`, `runbook`, `guide`, `gate`, `verification`. |
| `section_kind` | string | Section-level tag derived from markdown headings; see `SECTION_KIND_VALUES` in `weld/contract.py`. |
| `protocol` | string | Interaction-surface protocol: `http`, `grpc`, `event`, `ros2`, `inproc`. |
| `surface_kind` | string | `request_response`, `pub_sub`, `stream`, `one_way`. |
| `transport` | string | `tcp`, `http`, `http2`, `amqp`, `kafka`, `mqtt`, `ros2_dds`, `inproc`. Must be compatible with `protocol`; see `PROTOCOL_TRANSPORT_COMPATIBILITY`. |
| `boundary_kind` | string | `inbound`, `outbound`, `internal`. |
| `declared_in` | string | Source path where the interaction surface is declared. |

Producers follow an "omit instead of guess" rule: every closed-vocabulary
prop is optional, but once present it must be non-empty and drawn from
its allowed set. This keeps partial coverage honest.

## Edges

`edges` is a list of objects. Every entry has the following shape:

```json
{
  "from": "<source-node-id>",
  "to": "<target-node-id>",
  "type": "<edge-type>",
  "props": { ... }
}
```

### Required edge fields

| Field | Type | Description |
|---|---|---|
| `from` | string | Node id of the source; must exist in `nodes`. |
| `to` | string | Node id of the target; must exist in `nodes`. |
| `type` | string | Must be drawn from `VALID_EDGE_TYPES`. |
| `props` | object | Metadata bag. May be empty. |

Dangling endpoints are rejected by `validate_graph`. Fragment validation
(`validate_fragment(allow_dangling_edges=True)`) relaxes this for inputs
that will be stitched into a larger graph later.

### Edge type vocabulary

The set below reflects `VALID_EDGE_TYPES` as of contract version 5. The
canonical reference for governance and provenance additions is ADR 0016.

Core relationships:

- `contains`, `depends_on`, `produces`, `consumes`, `implements`,
  `documents`, `relates_to`.
- `responds_with`, `accepts`, `builds`, `orchestrates`, `invokes`,
  `configures`, `tests`.
- `represents`, `feeds_into`, `enforces`, `verifies`, `exposes`,
  `governs`.
- `calls` -- symbol-to-symbol call edge (see
  `weld/docs/adr/0004-call-graph-schema-extension.md`).

### Trace-participating vocabulary

The schema is broader than `wd trace`. A graph fragment can be valid and
renderable while still not appearing in trace output. Trace buckets:

- services: `service`, `package`
- interfaces: `rpc`, `channel`, ROS2 interaction nodes, or any node with
  recognized `protocol`, `surface_kind`, or `boundary_kind` metadata
- contracts: `contract`, `enum`
- boundaries: `boundary`, `entrypoint`, `compose`, `dockerfile`, `deploy`
- verifications: `test-target`, `test-suite`, `gate`

Trace walks these edge types in both directions: `accepts`, `builds`,
`calls`, `configures`, `consumes`, `contains`, `depends_on`, `documents`,
`enforces`, `exposes`, `feeds_into`, `implements`, `invokes`,
`orchestrates`, `produces`, `responds_with`, `tests`, and `verifies`.

`wd graph import` and `wd graph validate-fragment` warn when an imported
fragment has nodes or edges but none of them participate in this trace
contract. To make custom semantics traceable, map them onto these buckets
and edge labels at adapter output time.

Governance and provenance (ADR 0016):

- `owned_by` -- responsibility ownership.
- `gates` / `gated_by` -- bidirectional gate-to-target relationship.
- `supersedes` -- temporal replacement (new ADR over old ADR, new
  migration over old).
- `validates` -- validator-subject assertion, broader than `tests` or
  `verifies`.
- `generates` -- producer-artifact emission, distinct from build-pipeline
  `produces`.
- `migrates` -- data-model evolution between schema versions.
- `contracts` -- "agrees to honor" relationship between parties and
  interfaces.

Agent Graph (see ADR 0021):

- `uses_skill`, `uses_command`, `invokes_agent`, `handoff_to`,
  `references_file`, `applies_to_path`, `provides_tool`,
  `restricts_tool`, `triggers_on_event`, `overrides`, `duplicates`,
  `conflicts_with`, `implements_workflow`, `part_of_platform`,
  `generated_from`.

`relates_to` is the generic fallback. Strategies should prefer a
specific edge type when one fits -- collapsing distinctions into
`relates_to` erodes ranking, briefing, and context signals.

### Optional edge props

| Prop | Type | Notes |
|---|---|---|
| `source_strategy` | string | Strategy or agent that emitted the edge. |
| `confidence` | string | One of `definite`, `inferred`, `speculative`. |

Additional props are tolerated and round-tripped but not validated.

## Determinism and schema versioning

Per ADR 0012, `graph.json` is a deterministic artifact: two runs over
the same filesystem state produce byte-identical output. Every writer
goes through `weld.serializer.dumps_graph`, which enforces:

1. `nodes` emitted as a dict keyed by id, with keys in sorted order.
2. `edges` sorted by the tuple
   `(from, to, type, json.dumps(props, sort_keys=True))`.
3. All object keys serialized with `sort_keys=True` at every nesting
   level.
4. Fixed whitespace: `indent=2`, `ensure_ascii=False`, and a single
   trailing newline.

`meta.version` (contract vocabulary) and `meta.schema_version`
(federation layout) are independent. The federation layout version is
content-driven: it reports `2` when the graph contains any `repo:*`
node, `1` otherwise, so a rolled-back workspace returns to single-repo
byte parity with the pre-federation emitter.

Readers that only understand older federation layouts pass an explicit
`max_supported_schema_version` to `weld._graph_schema.load_graph_file`
and receive a `SchemaVersionError` on newer artifacts. The error
message names both the observed `schema_version` and the word
`upgrade` so old `wd` installs fail loudly instead of silently
misinterpreting federation constructs.

## Polyrepo federation

Under federation (ADR 0011), a workspace root holds a meta-graph that
aggregates across registered child repositories:

- Each child writes an ordinary `schema_version = 1` graph.
- The root emits a `repo:<name>` node per registered child, carrying
  `path`, `path_segments`, `depth`, and `tags`. Presence of any
  `repo:*` node stamps `schema_version = 2`.
- Cross-repo edges use node ids of the form
  `<child-name>\x1f<child-local-id>` (ASCII Unit Separator, `\x1f`) so
  endpoints remain unambiguous across repositories.
- Cross-repo resolvers declared in `.weld/workspaces.yaml`
  (`cross_repo_strategies`, e.g. `grpc_service_binding`,
  `compose_topology`, `service_graph`) run at root discovery time.

## Extension points

- **New node or edge types.** The contract vocabulary is strict by
  design. Adding a node or edge type requires an ADR, a contract
  `SCHEMA_VERSION` bump, and an entry in `VALID_NODE_TYPES` /
  `VALID_EDGE_TYPES` in `weld/contract.py`. Examples of prior
  extensions: ADR 0016 (governance edges), ADR 0021 (Agent Graph), and
  the call-graph extension in `weld/docs/adr/0004`.
- **New strategies.** Discovery strategies (config-driven via
  `.weld/discover.yaml`, with plugins in `weld/strategies/` or a
  project-local override in `.weld/strategies/`) emit nodes and edges
  using existing vocabulary. Strategies do not extend the vocabulary;
  they populate it. Their output is attributed through
  `props.source_strategy`.
- **New cross-repo resolvers.** Resolvers are declared in
  `.weld/workspaces.yaml` under `cross_repo_strategies`. The validator
  rejects unknown resolver names at load time so typos fail loudly.

## Compatibility policy

- **Additive change** -- new node type, new edge type, new optional
  prop, or new optional `meta` field. Requires a contract
  `SCHEMA_VERSION` bump and an ADR. Older readers receive an
  actionable validation error that names the observed version and the
  `wd discover` invocation needed to regenerate.
- **Breaking change** -- renaming or removing an existing vocabulary
  member, tightening a required field, or changing an existing
  prop's type. Treated the same as an additive change from the
  contract's point of view (bump `SCHEMA_VERSION`, write an ADR) but
  must also document a migration path for existing `graph.json`
  artifacts and announce the change in the release changelog.
- **Federation layout change** -- any change to the shape of
  `repo:*` nodes, cross-repo id encoding, or root-vs-child emission
  rules. Requires a bump of `ROOT_FEDERATED_SCHEMA_VERSION` in
  `weld/_graph_schema.py` and a loud, upgrade-worded error from
  `load_graph_file` for older readers.

Validation is enforced at every ingest point: `wd graph validate`
(whole-graph), `wd graph validate-fragment` (strategy output or
adapter), and every `add-node` / `add-edge` call on the `Graph` API.

## Example

A minimal valid `graph.json`:

```json
{
  "meta": {
    "version": 5,
    "updated_at": "2026-04-24T12:00:00Z",
    "schema_version": 1
  },
  "nodes": {
    "service:billing": {
      "type": "service",
      "label": "Billing service",
      "props": {
        "source_strategy": "compose_topology",
        "authority": "canonical",
        "roles": ["implementation"]
      }
    },
    "doc:adr-0016": {
      "type": "doc",
      "label": "ADR 0016: Edge-type vocabulary",
      "props": {
        "doc_kind": "adr",
        "file": "docs/adrs/0016-edge-type-vocabulary-extension.md"
      }
    }
  },
  "edges": [
    {
      "from": "doc:adr-0016",
      "to": "service:billing",
      "type": "governs",
      "props": {
        "source_strategy": "adr_scanner",
        "confidence": "inferred"
      }
    }
  ]
}
```

A federated root graph additionally contains `repo:*` nodes and stamps
`meta.schema_version = 2`; cross-repo edges use `<repo>\x1f<local-id>`
on both endpoints.
